# 実習のルール

## ソースコードのフォーマット

本プロジェクトでは Google Java Style Guide に基づいたコーディング規約を採用しています。

### フォーマット基本ルール

#### インデント: スペース4個（タブ禁止）

```java
// OK
if (condition) {
    doSomething();
}

// NG（タブ使用 → スペースに置き換えること）
// if (condition) {
// →   doSomething();  ← この → がタブ1個を表す
// }
```

#### 行の長さ: 120文字以内

```java
// OK
String message = subscriber.getName() + " のサービス加入日は "
    + subscriber.getSubscriptionDate() + " です。";

// NG（120文字超）
String message = subscriber.getName() + " のサービス加入日は " + subscriber.getSubscriptionDate() + " です。適切な位置で折り返してください。";
```

#### ブレース: K&Rスタイル（エジプシャンブラケット）

```java
// OK（開きブレースは同じ行）
public void doSomething() {
    if (condition) {
        // ...
    } else {
        // ...
    }
}

// NG（開きブレースを次の行に置く）
public void doSomething()
{
    if (condition)
    {
        // ...
    }
}
```

#### ステートメント: 1行に1つ

```java
// OK
int x = 1;
int y = 2;

// NG（1行に複数のステートメント）
int x = 1; int y = 2;
```

### 命名規則

| 種類                       | 規則           | 例                    |
| -------------------------- | -------------- | --------------------- |
| パッケージ                 | 全小文字・連続 | `com.example.project` |
| クラス・インターフェース   | UpperCamelCase | `ServiceSubscriber`   |
| メソッド・フィールド・変数 | lowerCamelCase | `subscriberName`      |
| 定数                       | CONSTANT_CASE  | `MAX_SIZE`            |

### Javadoc

- `public` クラス・メソッドには必須
- 記述順: `@param` → `@return` → `@throws` → `@deprecated`

```java
// OK
/**
 * 加入者IDをもとに加入者情報を取得する。
 *
 * @param subscriberId 加入者ID
 * @return 加入者情報。存在しない場合は空の Optional
 * @throws IllegalArgumentException 加入者IDが null の場合
 */
public Optional<Subscriber> findById(Long subscriberId) {
    // ...
}

// NG（Javadoc なし）
public Optional<Subscriber> findById(Long subscriberId) {
    // ...
}
```

### プログラミングの実践

#### 継承したクラスでインターフェースのメソッドをオーバーライドするとき `@Override` を常に付与する

```java
// OK
@Override
public String toString() {
    return "Subscriber{id=" + id + "}";
}

// NG（@Override を省略）
public String toString() {
    return "Subscriber{id=" + id + "}";
}
```

#### キャッチした例外は適切に処理する（無視しない）

```java
// OK
try {
    process();
} catch (IOException e) {
    log.error("処理中にエラーが発生しました", e);
    throw new ServiceException("処理に失敗しました", e);
}

// NG（例外を握りつぶす）
try {
    process();
} catch (IOException e) {
    // 何もしない
}
```

#### `static` メンバーへのアクセスはクラス名経由で行う

```java
// OK
int max = Integer.MAX_VALUE;
Math.max(a, b);

// NG（インスタンス経由でアクセス）
Integer i = 0;
int max = i.MAX_VALUE;
```

## AI利用のルール

- AIエージェントはあなたのアドバイザー役として動作します。
- プログラムの解説、作成する上のヒント、テストデータの作成、チェックリストの作成の補助を行います。
- 本AIエージェントは本研修にのみ利用できます。

## テストケース

`extra/report`フォルダに testcase_No<課題番号>.md を作成し、この内容をもとにプログラムをチェックします。

チェックリストの書式

```markdown
# <課題名> テストケース

## <確認内容(正常系)>

<どのような内容の確認か>

- [x] <チェック項目1>
- [ ] <チェック項目2>
- [ ] ...

## <確認内容(異常系)>

<どのような内容の確認か>

- [x] <チェック項目1>
- [ ] <チェック項目2>
- [ ] ...
```

## この方式を採用する意図

この形式には、従来のExcel管理と比較して以下のメリットがあり、本事業の「モダンな開発体験」および「AI活用の効率化」に直結します。

- **AIが読み書きしやすい（自動評価の実現）:**
  - Excelファイルは構造が複雑で、AI（ChatGPTやGitHub Copilotなど）に内容を理解させたり、結果を判定させたりするのが困難です。
  - Markdownであれば純粋なテキストデータであるため、AIに「このチェックリストの項目を網羅するテストデータを作成せよ」や「ログファイルと照合してチェックボックスを埋めよ」といった指示が容易に出せます。
- **Git/GitHubとの親和性:**
  - 研修のポイントとして「ソースコードのレビューはGitHubを経由した方法で行う」とあります。
  - Markdownチェックリストであれば、ソースコードと同じリポジトリでバージョン管理でき、講師はブラウザ（GitHub）上でチェックリストとコード、実行ログを並べてレビューできます。
- **進捗の可視化:**
  - GitHubのIssueやPull Requestの本文にこのテキストを貼り付けると、[ ] がインタラクティブなチェックボックスとして表示され、進捗状況（例：2/4 完了）がプログレスバーとして可視化されます。

## 運用イメージ

受講生には以下のフローで作業させます。

1. VS Code上で testcase_No<課題版号>.md を作成し、仕様書からテスト項目を書き出す。
1. プログラムを実行し、ターミナルに出力されたログや出力ファイルを確認する。
1. 確認できた項目の [ ] に x を入れて [x] にする（チェック済みとする）。
1. このファイルをソースコードと共にGitHubへコミット・プッシュする。
1. このように、「Excelという事務作業」を「エンジニアリング（Markdown記述）」に置き換えることで、学習のモチベーション維持と効率化を狙う施策となります。

## レビューのルール

### レビューの目的

- コードの動作を理解しているか確認する。
- コードの品質（可読性、保守性、効率性）の向上を図る。
- プログラムが要求仕様を正確に満たしているかを確認する。
- 潜在的なバグや設計上の問題を早期に発見し、修正する。
- 講師(上長)の承認を得るという行為を訓練する。

### レビュー担当者（レビュアー）の役割

- **建設的なフィードバック:** 改善点を明確かつ具体的に伝え、解決策の提案も検討する。
- **仕様との照合:** プログラムが仕様書通りに実装されているかを確認する。
- **テスト内容の確認:** 提出されたCHECKLIST.mdを確認し、テスト項目が適切かつ網羅的か、またテスト結果が正しく記録されているかを評価する。
- **効率性:** 非効率な処理がないかを確認する。

### レビューを受ける側（レビュイー）の役割

- **明確な説明:** レビュー依頼時に、変更の目的、実装の意図、懸念点などを明確に説明する。
- **必要情報の提供:** 関連する仕様書、テストデータ、テスト結果（CHECKLIST.mdを含む）などを準備し、レビュアーがレビューしやすいように提供する。
- **フィードバックの検討:** レビュアーからの指摘や提案を真摯に受け止め、改善策を検討し、適切に対応する。
- **質問:** 不明な点や疑問点があれば、積極的に質問し、理解を深める。

### レビューのプロセスとツール

- **Pushベース:** Git/GitHubにプッシュし、それを連絡する事でレビューを行うことを基本とする。
- **コメントと議論:** GitHubのコメント機能や、必要に応じて対面またはオンライン会議を活用し、議論を深める。
- **完了の判断:** レビュアーとレビュイー間で全ての指摘が解決し、コードが承認された時点でレビュー完了とする。

### コミュニケーションガイドライン

- **コードへの言及:** フィードバックは、コードの具体的な内容に対して行い、個人攻撃にならないよう注意する。
- **尊重の姿勢:** 異なる意見がある場合でも、互いの意見を尊重し、建設的な議論を心がける。
